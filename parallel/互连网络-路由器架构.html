基本路由器架构

上述为虚拟信道路由，模块可以大致分为两组：数据路线(datapath)和控制面板(control plane)。

首先包裹的每个flit到达路由的输入单元。输入单元有5种状态。

为了发送一个包裹，路由首先计算输出端口。给定输出端口后，从虚拟信道分配器中建立一条虚拟信道。上两步完成后，包裹的每个flit被交换器分配器分配一段时间空当，并在这段时间片内传输到输出单元。输出单元有3种状态。

路由器的控制在两个频率下工作：包裹传输率和flit传输率。路由计算和虚拟信道分配每包裹进行一次；交换器分配是每flit进行一次。

下面是路由运行流水线。RC—路由计算，VA—虚拟信道分配，SA—交换器分配，ST—交换传输。

阻塞(stalls)

阻塞有两种类型：包裹阻塞，阻止G变为R/V/A状态。flit阻塞指包裹在A状态，但是无法完成交换分配。

重新分配信道

当尾flit离开后，虚拟信道可以被循环重新分配给新的包裹。

有两种时候可以重分配输出信道x：当包A的尾flit进入ST时，将x分配给B，这种方法会产生由B至A的依赖，易产生死锁；或者当包A的尾flit进入下游ST时，将x分配给B，这样就不会产生依赖。

投机

可以通过投机的方法，来让不同阶段并行的执行，从而减少流水线长度，降低延迟。

如可以投机地将VA和SA阶段并行的执行，只有当SA和VA都没有被阻塞，才继续下一个阶段。对于没有VA阶段的主体flit，则执行非投机分配。同时，为了保证性能，还可以给予非投机分配更高的优先级。

当交换机有足够的加速比时，甚至可以进一步投机，将VA, SA, ST三步并行执行。

除此之外，还可以将RC这一阶段提前，在第i步时，就计算第i+1跳的RC，称为NRC(next-hop routing computation)。

flit和credit编码

为了区分flit和credit，最简单的方法就是通过不同的信道发送。更高效的方法是每发送1个flit，附上1个credit。这叫做肩负(piggyback)。

    (a)方案虽然给每个端口的存储分类带来灵活性，但是存储器的输入输出带宽会成为瓶颈。

    (b)方案很好的解决了带宽和延迟问题，但是虚拟信道只能对应一条物理信道。

    (c)方案可以在一条物理信道上建立多条虚拟信道。

  2. 输入缓存数据结构

    常用两种方式：循环缓冲器和链表。循环缓冲器需要将缓冲区进行固定的分割，而链表的开销较大。

  3. 输入缓存的分配

    动态分配可以将资源分配给较忙的虚拟信道，但这容易造成死锁。可以通过维持一个额外的计数寄存器来避免这种情况。

    计数寄存器可以分为计算保留单元的数目和浮动单元的数目。每个虚拟信道都维持一个保留单元的计数，而浮动单元技术是共享的。通过这个计数器可以执行不同的策略来避免死锁。

交换器

交换器一个重要参数是加速比。加速比是提供的交换带宽比上支持最大吞吐量的交换带宽。

  1. 总线交换

    当一个flit时间包含的phit时间个数大于端口数，那么一条总线可以被用作一个交换器。假设一个有P个端口的输入端接收到累计P个phit后，申请总线并传输P个phit，在输出端口分解这P个phit并一次性向下游结构传输。

    总线结构的好处是简单，但是会浪费端口带宽并增加延迟。主要花费在解串行化(deserializer)和串行化(serializer)上。

  2. 交叉开关

    假设一个k输入k输出路由器，有输入加速比为si，输出加速比为so，那么交叉开关有sik个输入。对于随机交换分配，吞吐量为在sk个flit种至少1个被选入到给定的输出的概率：

  3. 网络交换

    多个交换器组成一个交换网络，这可以减少交叉点的数量并且增强局部性。

输出结构

输出结构可以采取FIFO缓冲器，一个共享的FIFO缓冲器可以在所有虚拟信道共享且不会产生依赖，因为输出FIFO不会阻塞。回压不会影响到输出缓冲器，而是在SA阶段营销输入侧。然而，当输出缓冲器占用超过一定阈值后，可以向交换分配器发出回压信号来减少通信量。

仲裁器

仲裁时机

一些仲裁器仅仅授予1个时间周期，或者固定时间周期；一些仲裁器一直授予到资源被释放，有些是等到保持线信号为低电平时释放，有些是等到请求线信号为低电平时释放。

公平性

弱公平性：每个请求都均匀的服务

强公平性：每个请求都被以相同频率服务

FIFO公平性：每个请求按先后次序服务

注意即使仲裁器是局部公平的也未必是全局公平的。

可变优先级的仲裁器

上述仲裁器极不公平，可以引入独热(one-hot)优先级信号p来决定优先级：

  1. 无意识仲裁(oblivious arbiter)

    无意识仲裁的p的生成不利用r和g的信息，只是用一个移位器每个周期循环一个位。这无法产生强公平性。假设只有ri和ri+1是活跃的，而ri+1只在pi+1为真时，才能获得资源，其他n-1时间内都由ri获得资源。

  2. 循环仲裁(round-rubin arbiter)

    循环仲裁认为刚刚得到资源的请求应该拥有最低的优先级。

  3. 授予保持仲裁(grant-hold arbiter)

    上面的方法都是每次授予都只给予一个周期。授予保持可以利用添加保持信号，授予多个周期。

  4. 带权循环仲裁(weighted round-rubin arbiter)

    有时需要带有一定的不公平性，如请求者i带权wi，那么fi=wi/W部分的授权需要给请求者i。可以给普通循环仲裁加一个计数器，当计数器值减小到0时，降低优先级。

矩阵仲裁器

矩阵仲裁器执行的是最近最少服务准则(least recently served)，通过维护一个状态量wij来表示i的优先级是否大于j的优先级，且wij=¬wji, ∀i≠j。当第k请求获得授权后，清空第k行wk*，并设置第k列w*k，因为k是最近服务的。

队列仲裁器

队列仲裁器给每个请求一个时间戳，每次处理时间戳最早的请求。

时间戳的宽度可以用时间戳范围Δt和到达时间间隔来ta确定

时间戳范围必须足够大，设最大服务时间为Tmax：

当时间戳从最大滚动到0时，可以采用改变最大比特位的方法(most significant bit, MSB)。如假设Δt=16，ta=1，nTmax=8。那么，当时间戳从15到0时，所有时间戳都位于[8, 15]，因此MSB可以将[8, 15]→[0, 7]。当7滚动到8时，再一次改变MSB。

分配器

分配器接受n个m比特的输入，输出n个m比特。

并且遵循三天原则：

  • gij⇒rij，授予只能给对应的请求
  • gij⇒¬gik ∀k≠j，每类型输入请求只能有一个被授予
  • gij⇒¬gkj ∀k≠i，每类型输出请求只能有一个被授予

可以用n×m矩阵表示R和G。G中有最大可能的分配称为最大匹配(maximum matching)，即G中含有max(n, m)个1。理论上说，为了找到最大匹配，需要回溯，先暂时将1个位定为1，再在后续移除。但这样的算法复杂性高，因此常常使用启发式算法。

准确算法

可以用一个二分图(bipartite graph)来表示请求矩阵。通过这个图，可以构造一个有向残差图(directed residual graph)，图里若有边是有匹配的，那么从输出指向输入；若边还没有匹配的，那么从输入指向输出。构造之后，若能找到从一个未匹配输入到未匹配输出的一条有向路径，称之为增强路径(augmenting path)，那么就更新原图，将这条增强路径上的输入指向输出的路径变为输出指向输入(未匹配变为匹配)，将输出指向输入的路径变为输入指向输出(匹配变为未匹配)，这个过程一直重复到没有增强路径为止。

  1. 平行迭代匹配(Parallel Iterative Matching, PIM)

    通过随机可分分配的多次迭代，每个仲裁器在每次迭代执行随机优先级。

  2. iSLIP

    使用循环仲裁器，并且只在仲裁器产生了一个成功的授权时，才更新优先级。

  3. 孤单输出分配器(Lonely Output Allocator)

    加入一阶计算每个输出的请求数量，得到最少请求数量的输出，给出更高的优先级。

波面分配器

不像上一个分配器，波面分配器同时仲裁输入和输出。

一个有行(列)令牌的，但不能使用令牌的单元，继续向右(下)传输令牌。令牌的传播是按照如同波面的对角组优先级的形式。如果一个请求单元同时得到行和列令牌，则授权并停止传播令牌。

增量 vs. 批量分配

在许多路由中，分配器都是每隔i个周期工作，也就是说可以运行i个迭代次数的分配算法，这种称为批量分配(batch)。

而在增量(incremental)分配中，仅仅一个周期的分配器被用来产生最佳的分配。一旦收到授权立即传输flit。

多级分配(multistage)

一些应用需要一个路由先为高优先级的分配授权，再将剩下资源分配给低优先级。

网络接口(network interface)

处理器-网络接口

一个好的消息传递接口需要：1. 提供到网络的低开销；2. 不和其他处理器相互影响。

  1. 二寄存器接口(Two-Register Interface)

    有1个网络输出寄存器(network output register)，消息从这里发出，和1个网络输入寄存器(network input register)，消息从这里接收。

    缺点：1. 长消息而言开销过大；2. 不能从错误的软件行为中保护网络，有些消息在发送一部分后，过很长时间还无法发送下一部分。

  2. 寄存器映射接口(Register-Mapped Interface)

    解决安全问题，可以让消息自动从处理器一般用寄存器中发送。

  3. 描述符接口(Descriptor-Based Interface)

    解决2中长消息问题

  4. 消息接收(Message Reception)

    可以采取多线程的方式来接收消息。

共享内存接口

处理器-网络接口：当一个请求在高速缓存中缺失，那么就交给缺失状态保持寄存器(miss-status holding register, MSHR)。

内存-网络接口：当消息从网络接收时，初始化交易状态保持寄存器(transaction status holding register, TSHR)。

线性结构接口(Line-Fabric Interface)

错误控制

失败模式(Failure Modes)和错误模式(Fault Modes)

失败模式是错误的物理原因，如信道内的高斯噪声等，而错误模式是描述失败模式导致的相关行为，并隐藏了不必要的复杂部分。

BER(bit error rate) SER(soft-error rate) MTBF(mean-time between failures) Byzantine error：系统继续执行但是有着严重错误。

有些部分常常在开始和结束时，错误率较高

错误控制分为3个部分：检测(detection)、抑制(containment)、恢复(recovery)。

链路层错误控制

  1. 链路监控

    通过增加错误控制码(error control code, ECC)来检测错误。通常使用CRC。

    链路监控应该是连续的，否则当一个空闲信道没有被检测时，错误要一直到下一个flit来到时才能检测出。因此空闲信道可以发送空闲flit来让监控器检测。

  2. 链路重传(retransmission)

    为了向路由逻辑隐藏错误，隐藏包括抑制和恢复错误。重传是隐藏的最容易的方法。也可以使用向前错误订正码(forward error-correcting, FEC)。

  3. 信道重构，降级，关闭(channel reconfiguration, degradation, shutdown)

    当一个信道有重复的错误，或者BER比预期要高。、

    重构如信道提供一个额外的比特位，当一条信号线出错，可以使用另一条。或者减少传输率，如8-bit信道中有一条信号线出错，可以按7/8的原速率传送。若无法重构，则关闭信道。

路由错误控制

通过复制路由逻辑来检测。一份主复制(master)，和一份用来与主复制比较用的影复制(shadow)。常用SECDED码(single error correcting, double error detecting)。检测也可以通过一致性检查，如协议一致性、信用一致性、状态一致性。为了抑制错误，常常把路由划分位错误抑制区域。通常路由内提供额外的设计来替代错误的模块，称为热交换(hot swapping)。

网络层错误控制

可以通过自适应路由或路由表来替换失效的链接。

端到端错误控制

可以在发送端维持一份复制，从而在出错情况下重传。将端到端错误控制部署在网络层、路由层、链路层，可以让互连网络实现任意的可靠性。

总线

总线有两种性质：广播(broadcast)和串行(serialization)。

物理上，总线可以是单条串行总线，也可以是完全的并行总线，通常总线位于两者之间。

总线运行的单位有周期(cycle)，消息(message)，事务(transaction)。

总线有外部有序(externally sequenced)和内部有序(internally sequenced)。在外部有序总线中，所有信号通过外部中心排序器。在内部有序总线中，各自产生自己的传输使能。

一个总线周期可以是同步的，也可以是异步的。

总线内部有序传输消息过程涉及仲裁、选址、传送和应答这几个步骤。仲裁选择哪个模块开始事务，达到仲裁的叫做bus master。选址选择接收模块，将发送地址与模块内地址一一对比，找到接收模块。传送模块将消息从发送送到接收端。

高性能总线协议

  1. 总线流水线

    为了提高总线利用率，可以采取流水线形式，将总线事务分为若干模块。AR确认仲裁器请求，ARB仲裁器做出决定，AG仲裁器的授权传给请求者，RQ地址发送给存储模块，P位访问存储器，RPLY数据返回给请求者。

  2. 分事务总线(split-transaction bus)

    进一步减少总线空闲，可以将事务分为两种：发送消息和回复消息。

    对比图，(a)为流水线，(b)为分事务

  3. 爆发消息(burst message)

    为了减少标头的开销，通常选择发送长消息而不是分开发送短消息来增加效率。为了增加长爆发消息的效率并且给高优先级的请求低延迟，通常采用可以中断长消息发送的形式，在高优先级短消息发送完后，再继续发送长消息。
